# Active Projects

## Agent Instructions

When reviewing this file to give Aaron a projects update:

1. Collect all `[in progress]` and `[started]` items. Collect all `[on deck]` items.
   - Treat `[backlog]` as the initial queue for new projects before they become `[on deck]`.
2. New projects start in `BACKLOG` unless Aaron specifically says otherwise, then move up through `ON_DECK` and `IN_PROGRESS`.
3. **If there are active or on-deck items:** present them in two short groups ("In progress" / "On deck"), then ask whether any on-deck items are ready to move to in progress.
   - Use this formatting:
     - `Daily projects check.`
     - Blank line
     - `📗 In progress:`
     - One item per line, with a period after each item name
     - Blank line
     - `📘 On deck:`
     - One item per line, with a semicolon after each item name except the final item
     - Blank line
     - `Are any of the on-deck items ready to move to in progress?`
4. **If both groups are empty:** suggest what Aaron could start next using this flow:
   - Pull all `[backlog]` items.
   - Glance at recent `[done]` items for natural follow-ons (e.g. a finished build that implies a cleanup or next phase).
   - Group similar items under a shared label — for example, GitHub repos or tech tools to try collapse into "try a new tech tool"; personal admin tasks can be grouped together; creative/maker projects can be grouped together. Use your judgment to find the right groupings.
   - Aim for 3–5 suggestions total. Keep the framing light: "nothing active right now — here are some things you could kick off."
5. When Aaron replies, update statuses in this file accordingly and log a dated update line on any item that changes.

- Status note: `started` means the same thing as `in progress`.
- Status note: `backlog` means queued but not yet ready to be on deck.

## Project Properties

Projects can have optional property lines directly below the status line (before `- Updates:`). Read these and apply judgment — they are hints, not rigid rules.

| Property | Meaning |
|----------|---------|
| `check_after: YYYY-MM-DD` | Don't ask for a status update on this project until on or after this date. |
